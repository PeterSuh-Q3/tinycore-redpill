# alpine-redpill v1.4.0.0

새로 만들어진 **alpine-redpill** 브랜치의 첫 정식 릴리즈입니다. 이번 버전부터 로더의 실행 환경이 TinyCore Linux(glibc) 기반에서 **Alpine Linux(musl)** 기반으로 이전됩니다.

`main` 브랜치의 `tinycore-redpill`은 **v1.3.1.1**에서 기능 업데이트를 멈춥니다(기존 v1.3.1.1 이하 사용자는 지금까지처럼 그 라인에서 자동 업데이트를 계속 받습니다). v1.4.0.0부터 새로 시작하는 사용자는 `alpine-redpill` 라인을 따라가게 됩니다.

<details>
<summary><b>왜 이미지 용량이 2GB/4GB에서 3GB/5GB로 바뀌었나요? (클릭해서 펼치기)</b></summary>

<br>

한 줄 요약: **4번째 파티션이 새로 추가되었고, 그만큼(1GB)이 그대로 늘어났습니다.**

기존 로더 이미지는 파티션 3개로 구성되어 있었습니다.

| # | 내용 |
|---|---|
| 1 | GRUB(BIOS + UEFI), `grub.cfg` |
| 2 | (변경 없음) |
| 3 | 예비 파티션(과거엔 TinyCore 런타임 파일이 있었으나, 이번엔 완전히 비워둠) |

이제 redpill 로더 자체를 빌드/재생성하려면 실제로 동작하는 Alpine Linux 환경이 필요합니다(예전엔 TinyCore 데스크톱 환경이 이 역할을 했습니다). 매번 인터넷 연결에 의존하는 대신, **4번째 파티션(라벨 `alpine`, 1024MB)**을 새로 추가해 다음을 담았습니다.

- `vmlinuz-lts` / `initramfs-lts` / `modloop-lts` — Alpine Linux 디스크리스 부팅 세트 전체
- `repo-main` / `repo-community` — Alpine 패키지 저장소의 오프라인 로컬 미러
- `localhost.apkovl.tar.gz` — 로더 빌드에 필요한 사용자/서비스/도구가 미리 구성된 환경 오버레이

GRUB의 새 메뉴 항목 **"Alpine Redpill Image Build"**를 선택하면 이 파티션으로 곧바로 부팅되어, 별도 USB나 인터넷 연결 없이도 완전히 독립적인 Alpine 빌드 환경을 바로 쓸 수 있습니다.

이 추가된 1GB 파티션이 용량이 늘어난 이유의 전부입니다.
- **2GB 버전 → 3GB** (기존 2GB 콘텐츠 + Alpine 파티션 1GB)
- **4GB 버전 → 5GB** (기존 4GB 콘텐츠 + Alpine 파티션 1GB)

기존 파티션들의 *용량* 자체는 달라진 게 없습니다 — 예전에 TinyCore 시절 파일들이 있던 3번 파티션은 이번엔 완전히 비워두었습니다(빌드 과정에서 더 이상 레거시 콘텐츠를 복사해 넣지 않음).

</details>

<details>
<summary><b>Alpine Linux로 옮기면서 얻는 이점은 무엇인가요? (클릭해서 펼치기)</b></summary>

<br>

- **훨씬 가벼운 용량.** Alpine은 musl libc + BusyBox 기반이라, 동등한 기능 대비 TinyCore의 glibc 기반 유저랜드보다 훨씬 작습니다.
- **현대적이고 빠른 패키지 관리.** TinyCore의 `.tcz` 확장 방식 대신 `apk`를 사용해, 제대로 된 의존성 해석과 훨씬 활발히 관리되는 패키지셋을 제공합니다.
- **활발한 업스트림 유지보수.** Alpine은 보안 패치와 커널 업데이트가 잦은 반면, TinyCore는 상대적으로 릴리즈 주기가 많이 느려진 상태였습니다.
- **최신 커널, 넓은 하드웨어 지원.** `-lts` 커널 라인을 사용해, 노후화된 TinyCore 커널 대비 최신 드라이버 지원을 기본으로 제공합니다.
- **더 깔끔한 초기화 시스템.** 여러 개의 임시방편적인 `bootlocal.sh`/`bootsync.sh` 셸 로직 대신 OpenRC 서비스 관리 체계를 사용해, 부팅 흐름을 이해하고 확장하기 쉬워졌습니다.
- **이 용도에 정확히 맞는 설정 모델.** Alpine의 `apkovl` 오버레이 + `lbu commit` 메커니즘은 애초에 디스크리스/라이브 시스템을 위해 설계된 것이라, 사용자/서비스 설정 전체가 파일 하나로 압축되는 방식이 로더의 "빌드 환경" 파티션 구조와 자연스럽게 맞아떨어집니다.
- **오프라인 패키지 복원 내장.** 새로 추가된 4번 파티션이 로컬 apk 미러 역할도 겸해서, 부팅 시 패키지 복원이 더 이상 인터넷 접속에 전적으로 의존하지 않습니다.
- **더 견고해진 하드웨어 자동 감지.** 이번 이전 작업의 일환으로, 디스크(`rebuildfstab`)에 이어 네트워크 인터페이스(`rebuildnetifaces`) 설정도 매 부팅마다 실제로 감지된 장치를 기준으로 동적으로 생성하도록 바뀌었습니다 — 고정된 하드코딩 목록에 의존하다 추가 디스크나 랜카드를 조용히 놓치던 방식에서 벗어났습니다.

</details>

## xTCRP 변형에 대해 — Alpine과 전혀 무관합니다

`xtcrp` 이미지 변형을 사용하신다면, **xTCRP는 이번 Alpine 이전 작업과 아무 관련이 없습니다.** 기존과 완전히 동일한 방식으로, 별도의 "friend" 커널(`bzImage-friend`/`initrd-friend`, Alpine과는 다른 Buildroot 기반 별도 프로젝트에서 빌드됨)을 로더에 함께 담아 빠른 설정용 부팅 경로를 제공하는 구조입니다. 이 메커니즘 자체는 손대지 않았고, 다른 것들과 함께 빌드 스크립트 버전만 v1.4.0.0으로 올라간 상태입니다. xTCRP는 Alpine Linux를 쓰지 않고, 4번째(`alpine`) 파티션도 없으며, 위에서 설명한 변경 사항들과 무관합니다.

<details>
<summary><b>그 외 함께 반영된 수정 사항 (클릭해서 펼치기)</b></summary>

<br>

- **듀얼랜(NIC 2개 이상) 부팅 지원.** 기존에는 부팅 시 처음 감지된 랜카드(`eth0`)만 자동으로 DHCP IP를 받고, 두 번째 이상의 랜카드(`eth1` 등)는 전혀 설정되지 않은 채로 남아있었습니다. 새로 추가된 부팅 스크립트(`rebuildnetifaces`, 기존 디스크 자동 감지 스크립트 `rebuildfstab`과 같은 방식)가 매 부팅마다 감지된 모든 네트워크 인터페이스를 스캔해 빠진 것을 자동으로 설정합니다 — 이제 랜카드가 여러 개인 장비도 기본적으로 모든 인터페이스에서 IP를 받습니다.
- **메뉴에서 ESC/Cancel을 눌러도 종료되지 않음.** 기존에는 메인 mshell 메뉴에서 실수로 Esc나 Cancel을 누르면 바로 셸로 빠져나갔습니다. 이제는 메뉴가 그대로 다시 표시되며, 종료는 메뉴의 명시적인 "Exit" 항목을 선택했을 때만 가능합니다.

</details>

## 참고

- 기존 v1.3.1.1 사용자는 영향받지 않으며, `main`/v1.3.x 라인에서 계속 업데이트를 받습니다.
- `master` 브랜치는 v1.3.1.1 시점의 영구 보호(protected) 스냅샷으로 다시 만들어, 아직 그 이름을 참조하는 구버전 클라이언트 코드도 안전하게 서빙되도록 했습니다.
